The internal architecture of
JARVIS & SKYNET.

Two symmetric AI orchestrators — one for Demand, one for Supply — built on a shared component model with externalized state, two vertical seams for cross-cutting concerns, and standardized open protocols for both peer communication and sub-agent invocation.
This document describes the internal architecture of the two top-level orchestrators in Project Phoenix. It first draws an important boundary (§3): not all work belongs in an orchestrator — fixed-sequence business pipelines run on a workflow engine, while only runtime-decided, conversational, or negotiated work runs through JARVIS and SKYNET. The rest of the document then details those orchestrators. Both follow a five-layer component model (input → cognition → execution → state → output) with externalized domain state. Two vertical seams cut across all layers: an A2A Protocol Handler that owns peer negotiation end-to-end, and a Governance Controls seam covering identity, kill switch, FinOps, human-in-the-loop, and observability. Peer communication uses the Linux Foundation's A2A v1.0 protocol; sub-agent invocation uses MCP. The architecture is engineered for EU AI Act high-risk compliance (enforceable August 2026) and includes an explicit evolution path toward the broader agentic mesh.

Key architectural commitment

JARVIS and SKYNET are structurally symmetric peers. Both publish Agent Cards. Both can issue and receive A2A Tasks. Neither is "in charge" of the other. This symmetry is expressed in industry-standard contracts (Agent Cards, A2A Tasks, MCP), which means the orchestrators can eventually federate with partner systems outside Phoenix without changing their internal architecture.

§ 2

Standards Alignment

Phoenix sits on top of three Linux Foundation / open standards that became production-ready in 2025-2026: A2A for peer communication, MCP for tool / sub-agent access, and Agent Cards for identity and discovery. The Phoenix-specific domain logic (FEAS payloads, supply-chain decisions) rides as the application layer on top.

2.1 Protocol Stack

Read top-down. Each layer has a single responsibility and is replaceable independently. Phoenix-specific concepts (FEAS_REQ payload structure, capacity reasoning) live at the top; everything below is industry-standard, vendor-neutral, and battle-tested by 150+ organizations adopting A2A.

L4 · App
Phoenix Domain Logic supply-chain semantics
Custom: FEAS_REQ / FEAS_RESP / FEAS_SELECT payloads; capacity reasoning; AFP gating; TC class priority. Carried inside A2A Task messages — A2A doesn't care what's in the payload, Phoenix does.
L3 · Coord
Coordination Protocols how agents talk and discover
Peer-to-peer: A2A v1.0 Task lifecycle (submitted → working → input-required → completed / failed / cancelled). Sub-agent calls: MCP JSON-RPC. Discovery: Agent Card at /.well-known/agent.json.
L2 · Transport
Wire Transport bytes on the wire
gRPC (primary), JSON-RPC 2.0, or REST — A2A v1.0 supports all three equally. Protobuf canonical schema (a2a.proto). Streaming for long-running tasks. Phoenix default: gRPC for orchestrator-to-orchestrator, JSON-RPC for orchestrator-to-MCP-server.
L1 · Id
Identity & Trust who is calling, are they allowed
OAuth 2.0 client credentials + JWS signed Agent Cards. Each agent has cryptographic identity. Each capability advertised in Agent Card includes auth requirements. Required by EU AI Act for "persistent identity management throughout the agent lifecycle".

Why this matters for Phoenix

Three reasons. Identity: when SKYNET receives a FEAS_REQ, it can verify it really came from JARVIS (not a spoofed agent) by checking the JWS signature against the published public key. § 3

Two Orchestration Layers

Not everything in Phoenix should go through JARVIS or SKYNET. The system has two distinct ways of coordinating work, and choosing the wrong one for a given task is the most common and most expensive design mistake. This section explains the difference, why it matters, and how to decide which layer a workflow belongs to.
The core principle

An AI orchestrator must earn its cost with value a workflow engine can't provide. Orchestrators are good at deciding at runtime — which agent to call, in what order, how to handle the unexpected — and at negotiating and understanding natural language. If a task doesn't need any of those, a workflow engine does it faster, cheaper, and more reliably. "A step uses an agent" does not mean "the task needs an orchestrator." What matters is whether the sequence itself is decided at runtime.

3.1 The Two Layers

Phoenix coordinates work in two fundamentally different ways. Both call the same sub-agents (ORACLE, APOLLO, T-800, …) through MCP — but they decide what runs when very differently.

Layer A · Business Pipelines

Fixed sequence, scheduled cadence

The steps and their order are known in advance. A workflow engine runs them on a schedule or trigger, calling agents where AI work is needed and pausing for humans where approval is needed.

  • Runner: workflow engine (Temporal / Camunda)
  • Trigger: schedule (monthly cycle) or event
  • Decision about sequence: made by the designer, encoded in a DAG
  • Cost: low — no LLM reasoning to pick the next step
  • Reproducibility: 100% — same input, same path
  • Examples: monthly forecast → SQ, weekly optimization, daily ATP refresh
Layer B · Conversational / Event Orchestration

Runtime-decided sequence, negotiation

The path depends on context that only exists at runtime. JARVIS and SKYNET decide which agents to call, react to failures by negotiating alternatives, and talk to humans in natural language.

  • Runner: JARVIS / SKYNET orchestrator
  • Trigger: user message, peer request, or deviation event
  • Decision about sequence: made at runtime by the orchestrator
  • Cost: higher — LLM reasoning each turn (justified by flexibility)
  • Reproducibility: acceptable variance — "gets smarter" over time
  • Examples: answer feasibility, negotiate alternatives on escalation, handle a deviation cascade

3.2 How to Decide — Two Questions

The choice comes down to two independent questions. The first decides the layer; the second only decides whether a step happens to use an agent — it does not push the task toward an orchestrator.

The key insight people miss

The two highlighted boxes both use AI agents. The difference is not "does it use AI" — it's "who decides the order." In an Agent Pipeline the order is fixed by the designer; in an Agent Orchestrator the order is decided at runtime. A monthly forecast cycle calls ORACLE, waits for DMO, calls the optimizer, writes the SQ — every time, in that order. That is a pipeline, even though every step is an agent. It does not belong in JARVIS.

3.4 Why Getting This Right Matters

Routing a fixed pipeline through an AI orchestrator is the most common over-engineering mistake, and it is costly on four fronts.

Concern What goes wrong if a pipeline runs through JARVIS / SKYNET
Cost Every monthly cycle pays for LLM tokens to "decide" a sequence that was never in doubt. The order is fixed — paying a model to re-derive it each run is burning money for nothing.
Determinism DMO must be able to explain why this month's optimization produced these numbers, and reproduce them. A workflow engine gives byte-identical reproducibility; an LLM-driven path introduces variance that is hard to defend in a review.
Audit & compliance Workflow engines carry audit patterns that have passed financial audits for two decades (Camunda's lineage is in banking). There's no need to reinvent this inside an orchestrator's audit log — which exists for AI decisions, not every state change.
Operational maturity DevOps / SRE teams already run Temporal / Airflow well — monitoring, replay, debugging are mature. AI orchestrators need scarcer specialist skills; don't spend that scarcity on work a workflow engine handles.
Six-question test — run it on every workflow before building

If the answer is "yes" to all six, the task does not need an orchestrator — build it as Layer A: (1) Do you know every step before it starts? (2) Can every decision be written as if/else or SQL? (3) Is the input structured data rather than natural language? (4) Is there no mid-flow human clarification? (5) Can failures be handled by retry / DLQ / manual review? (6) Is there no need to explain the result in natural language?   Two or more "no" answers → an orchestrator (Layer B) probably earns its place.

3.5 What a Pipeline Tells the Orchestrators

A natural question when a pipeline runs without JARVIS or SKYNET in the loop: what, if anything, does it need to tell them? The answer follows one principle — a pipeline announces its committed results (facts) to the outside world; it does not report its intermediate progress to an orchestrator. JARVIS doesn't want to know which step the pipeline reached; it wants to know what changed that it might need to answer a user about.

Three destinations, not one

Pipeline data fans out to three different places depending on what kind of data it is. Note that none of them is "write directly into the orchestrator's memory" — the orchestrator obtains what it needs by reading a Data Product (pull) or by subscribing to an event (push it initiates), never by the pipeline injecting state into it.

Kind of data Where it goes Owner
Business result DP Data Product (Dolphin) Pipeline writes · everyone reads
Pipeline's own audit Workflow engine history Workflow engine
"Something happened" event Event bus (Kafka) Pipeline emits · orchestrator subscribes

What the orchestrator must know — and what it must not be told

Send (via event + DP)

Facts the orchestrator can act on

  • SQ committed — emit sq.published with light metadata (cycle, version, effective date, count of changed BU/grades). Not the SQ payload itself — just "new data is in Dolphin, go read it." JARVIS uses this to notify Sales and invalidate any cache referencing the old SQ.
  • Business-affecting failure — optimization infeasible, or DMO approval missed its deadline. Emit forecast_cycle.failed so SKYNET knows capacity planning may be on stale data and JARVIS knows how to answer quota questions.
  • Status frequently asked about — if Sales often asks "where's this cycle at?", expose status through a pipeline API the orchestrator can query — not pushed into memory.
Don't send (keep in pipeline)

Data the orchestrator can't use

  • Step-by-step progress ("now on step 3/6")
  • Intermediate values (draft forecast before DMO adjusts)
  • Retry attempts and technical errors it recovers from itself
  • Which button DMO pressed when (workflow history is enough)
  • Every row ORACLE computed (that lives in the DP)

The orchestrator can do nothing useful with these. If a user truly asks, query the workflow engine then — keep data in one place, with its owner, rather than copying it into two.

The mechanism — event + pull, never push-into-memory

The connection uses three loosely-coupled paths. The pipeline emits an event and continues immediately — it never waits for the orchestrator to acknowledge. If JARVIS happens to be down at that moment, the pipeline still finishes; the event waits in the bus for JARVIS to replay later. This is exactly why the pipeline must not call the orchestrator synchronously: that would make Layer A depend on Layer B, the wrong direction.

The event is a pointer, not a payload

Keep the event light. It carries enough to locate the result and decide whether to act — not the result itself. A fat event creates a second copy of the data that drifts out of sync with the DP.

{
  "event":        "sq.published",

  "emitted_at":   "2026-03-07T09:12:04Z",

// then read dolphin://sq/2026-04/v3 only if/when a user actually asks.
Three lines not to cross

1. Don't let the orchestrator's audit log become the pipeline's history. Pulling every pipeline step into the orchestrator audit "to have one place" bloats it, slows queries, and 99% is never read. The orchestrator audit is for AI decisions (why JARVIS chose this alternative), not every workflow state transition.

2. Don't let JARVIS trigger pipelines on the normal path. The monthly forecast fires from the workflow engine's own scheduler. JARVIS triggering a run should be an exception only (DMO asks for a re-run via chat), never the main route.

3. Don't let the pipeline block on the orchestrator. Emit the event and move on. If the pipeline waited for JARVIS to acknowledge, Layer A would depend on Layer B — a failed or slow orchestrator would stall business-critical pipelines.

3.6 How the Two Layers Connect

Putting it together: the layers share the same sub-agents through MCP, and the orchestrators interact with pipelines through events and APIs rather than by running them. A healthy Phoenix has the most components in Layer A (pure workflow and agent pipelines), a smaller set of genuine orchestration intents in Layer B, and a single shared pool of agents underneath both.

Interaction How it works
Pipeline → Agent Workflow engine calls an agent's capability via MCP at an AI step (e.g. ORACLE.forecast). Same MCP server the orchestrator would call.
Orchestrator → Agent JARVIS / SKYNET call the same agents via MCP during a conversation or event flow. Agents don't know or care who called them.
Orchestrator → Pipeline JARVIS triggers a pipeline re-run via the workflow engine's API ("DMO asked to re-run optimization"), or reads a pipeline's output (the SQ) from the DP — but does not execute the steps itself, and only on an exception path.
Pipeline → Orchestrator The pipeline emits a light event (e.g. sq.published) the orchestrator subscribes to, so JARVIS can notify Sales and refresh caches. The pipeline never calls the orchestrator inline or waits for it.

3.7 Inquiry Intake — the Layer-A Path (where SAGE belongs)

Opening an inquiry is the highest-frequency action in the system and is almost entirely deterministic — clone a prior line, adjust quantity / price / date, validate, submit. By the rule in §3.2 it therefore belongs in Layer A: it runs as a pipeline / UI action that calls the SAGE skill over MCP, with no orchestrator reasoning and no LLM on the path. After the inquiry is OPEN the same Layer-A pipeline continues — gate checks (CBS & AFP-floor, SQ, credit) and a direct T-800 can-commit — with orchestration (JARVIS ↔ SKYNET FEAS) entering only on escalation; both paths are shown in §8.1.

What changed from the earlier draft

An earlier version routed "open inquiry" through the JARVIS reasoning loop and treated it as a Layer-B conversational decision. That is corrected here: intake is a deterministic Layer-A skill (SAGE); the post-OPEN checks are likewise Layer A, and the orchestrator engages only when T-800 cannot commit. FIG 8.1 is reframed accordingly — it depicts the post-OPEN hybrid fast path (and its escalation), not the act of opening the inquiry.

SAGE is a Layer-A skill, not a reasoning sub-agent. Its duties — search, silent clone, completeness check, validation, submit — are rule-based; the only "smart" step (warn when a grade / package was never sold to this customer) is a lookup plus a rule, not LLM judgment. SAGE is therefore exposed over MCP and invoked by the intake pipeline — not gated behind JARVIS. JARVIS contributes to intake only at the optional conversational front door (natural language → structured fields), never as a step the inquiry must wait on.

Two deterministic gates precede a successful open, both rule-based (no LLM): a completeness gate (missing fields are re-requested in a loop) and a sold-before gate (an exact match on customer + grade + package must exist; if not, SAGE warns and requires explicit confirmation before submitting). Silent auto-match means the system itself pulls the latest exact-match record as defaults and overlays the user's input — there is no "pick one of three" step. Because the match key and the sold-before check are the same lookup, a miss is simultaneously the signal "never sold this combination" and the trigger for the warning.

Latency — solved by construction (hybrid fast path)

Cost and latency come from invoking an LLM, not from passing through pipeline logic. The post-OPEN path is not a bottleneck because the happy path never invokes an orchestrator at all: CBS & AFP-floor are ORACLE lookups, the SQ gates an APOLLO check, credit a WALLET query, and the commit a direct T-800 can-commit call — all driven by the workflow engine, no LLM on the path. It is also asynchronous to the user (Sales is already confirmed by intake), and LLM-only on judgment (FEAS negotiation fires only for the hard cases — shortfall, counter-proposal, ambiguity, Black high-value). That fast-path / slow-path split is what §8.1 shows; the genuine production risk is context / state consistency (§8 onward), not latency.

Ownership splits by data domain. ORACLE owns customer classification (CBS) — it computes Black/White, but because the value changes rarely the result is stored on the Customer master and read back at inquiry — and owns all price logic (AFP / price-floor). Summary

Phoenix has two ways to coordinate work. Business pipelines (monthly forecast, weekly optimization, daily ATP refresh) have a fixed sequence and run on a workflow engine that calls agents through MCP — opening an inquiry is one such Layer-A pipeline (§3.7). Conversational and event-driven decisions (answer feasibility, handle deviations, and the orchestration that follows an opened inquiry) have a runtime-decided sequence and run through JARVIS / SKYNET. Both share the same agents. When a pipeline finishes it announces the result with a light event and writes it to the DP — it never pushes state into the orchestrator or waits for it. The forecast-to-quota flow is a pipeline; it does not belong in the orchestrator. The rest of this document describes Layer B, the orchestration layer, since that is the part unique to Phoenix; the pipelines use standard, well-understood workflow tooling.

Why WALLET is SKYNET-home even though JARVIS calls it often

"Home side" answers who owns it — its objective, its spec, and which orchestrator sequences it on escalation — not who may call it. Calling across sides is always allowed (the assistant pulls a credit balance for Sales; the happy-path workflow calls WALLET as a gate). WALLET's home is SKYNET because a credit check is a feasibility veto (pass/fail, a hard constraint — §7.1), and in the redesigned process flow credit sits in the Operational level alongside Inquiry → Bid-Order, which maps to SKYNET (operational), not JARVIS (tactical). So Sales reaching WALLET through the JARVIS assistant is a cross-side call, while WALLET still belongs to — and answers in the feasibility terms of — SKYNET.

4.3 One Server, Two Callers

A sub-agent is an MCP server — a callee. What differs is the caller. A Layer-A pipeline's workflow engine calls ORACLE.forecast on a fixed schedule; JARVIS calls the same tool mid-conversation when a Reasoner step needs it. The capability is written once and the server is unaware of — and indifferent to — which caller invoked it.

Two consequences follow, and both are load-bearing for the rest of this architecture. First, sub-agents never call each other: there is no agent-to-agent fan-out at this layer, only caller→callee edges. Second, sequencing always lives with the caller — the workflow engine owns it for a Layer-A pipeline (encoded in a DAG), and the orchestrator owns it for Layer-B work (decided at runtime). The MCP server contributes a capability, never a sequence. This is why the same ten sub-agents can be shared across both orchestration layers without duplicating logic, and why adding a caller never requires touching a sub-agent.

Peer Protocol — A2A v1.0

Peer negotiation between JARVIS and SKYNET uses the Linux Foundation A2A v1.0 protocol. Phoenix-specific FEAS payloads ride inside A2A Task messages. A2A handles transport, identity, deadlines, and lifecycle; Phoenix defines the domain content.

7.1 The Decision Model — Who Decides What

The most important rule of the peer relationship: the two orchestrators do not optimize one shared score. They hold two different objective functions and meet through a bounded negotiation.

  • SKYNET decides feasibility — and can veto. "Can this physically be done, and at what cost-to-serve?" Inputs: ATP date + gap, shortfall (MT), capacity cap %, inventory zone (DOM/EXP), next production date + transition cost, cost-to-serve, credit (via WALLET).
  • JARVIS decides desirability — the commercial choice. "Of the feasible options, which is best for the customer / business?" Inputs: AFP gap (price vs. floor), tier / CBS, SQ / allocation remaining, required date, margin / deal value, credit headroom, opportunity cost.

Hard always beats soft. A hard constraint (supply or credit) is a wall; a soft preference is a want. Negotiation happens only inside the soft zone — date, split, package, partial quantity. If a request is infeasible on a hard constraint, no commercial preference changes the answer; the path is a feasible alternative (soft zone) or escalation.

Design rationale — why not one shared score

The obvious alternative is a single weighted score both orchestrators optimize together, e.g. score = w₁·margin + w₂·feasibility + w₃·priority − w₄·cost. It is rejected for four reasons:

  1. It conflates "can't" with "don't prefer." Feasibility is a hard fact — stock absent or capacity full means the order cannot ship; margin and date are wants. A weighted sum lets a high-margin deal out-score an infeasibility and "buy past" the wall — committing something that cannot be delivered. Hard constraints must be lexicographic (feasibility decided before preference), not a weighted sum where money outweighs a penalty.
  2. It forces one agent to own the other's domain. Whoever computes the blend must model both sides — JARVIS would need inventory zones / caps / transition cost, or SKYNET would need margin / tier / AFP. Two objectives keep each domain with its owner and the peers symmetric.
  3. It hides the very trade-off a human is there to resolve. Cases escalate because dimensions conflict (serve Black on-time vs. displace White; better margin vs. a costly replan). Collapsed to score = 0.82, the number has erased what the human needs. Two objectives let the escalation brief show commercial │ supply side by side (§10.5).
  4. Nothing actually needs the sum. When feasible options collide, a deterministic lexicographic tiebreak (Appendix 11.2) separates them reproducibly; the rare unresolved case goes to a human. No combined score is required at any point.

7.2 The FEAS Negotiation Protocol

FEAS is a three-message exchange, deadline-bounded and usually settled in one round.

  1. FEAS_REQ (JARVIS → SKYNET) — "Can you commit this exact line?" Carries the demand, the commercial context SKYNET needs for gating / triage, and the soft-zone bounds (how far the customer will flex).
  2. FEAS_RESP (SKYNET → JARVIS) — one of: ACCEPT (feasible exactly as asked); COUNTER (infeasible as asked → N structured alternatives, each pre-checked against hard constraints); INSUFFICIENT (no feasible alternative in the horizon → reason + earliest recovery date).
  3. FEAS_SELECT (JARVIS → SKYNET) — JARVIS's Reasoner ranks the alternatives (optionally confirming with Sales via the Dialog Manager) and selects one. SKYNET commits (hard-lock + schedule) and returns an ACK.

Properties. Bounded — typically a single round. Deadline-enforced — each FEAS_REQ carries a deadline; SKYNET does not block past it, and an unmet deadline is itself an escalation trigger. HITL pause — if a human decision is required, the A2A Task enters input-required. COUNTER is feasibility-bounded — every alternative SKYNET offers has already passed the hard-constraint cascade (T-800 → TEMPO → SIGMA); JARVIS never re-checks feasibility, it only ranks desirability.

7.5 Arbitration — No Central Referee

There is no central arbiter agent sitting between the peers. Conflicts are resolved by three layers, in order:

  1. A shared, a-priori contract. Both orchestrators load the same FEAS schema, hard-constraint definitions, and deadline policy. The "referee" is embedded in the rules both sides already implement.
  2. A deterministic tiebreak policy. When options collide, resolution is rule-based and reproducible — applied top-down, the first rule that separates two options wins (full lexicographic order: Appendix 11.2).
  3. The human (HITL) — exception only. DMO / SE / BU Manager are invoked only when there is no feasible option, the action exceeds an authority limit, or the deadline passes.
FIG 7.5 Conflict resolution — shared contract → deterministic tiebreak → human (exception only)
Design rationale — why no central arbiter

A natural-looking alternative is a third agent — ARBITER — that takes JARVIS's request, asks SKYNET if it is feasible, and decides. It is rejected for five reasons:

  1. It re-centralizes both domains. To decide, it must know JARVIS's commercial logic and SKYNET's supply logic; all logic piles into one agent while the orchestrators degrade into data pipes, and every rule change must be made in three places. Domain belongs with its owner. (This is the topology-level version of the "one shared score" problem in 6.1.)
  2. It breaks peer symmetry. The pack is built on two equal peers over A2A; a central arbiter turns that into hub-and-spoke and contradicts the reason A2A was chosen — you would lose each side's freedom to evolve and redeploy independently.
  3. Single point of failure and bottleneck. The arbiter sits on the critical path of every negotiation: down → nothing completes; slow → everything is slow. Peers talking directly contain failures to a component.
  4. It only moves the problem. The real question is how are two diverging sides reconciled? A box labelled "ARBITER" relocates the question; the arbiter still needs rules. If rules are needed anyway, put them where both sides read them (reason 5).
  5. What an arbiter should be is not an agent — it is "rules + a human."

The key shift: the deciding rules move earlier (an a-priori shared contract) and higher (a governed Policy Store, §10.3) instead of into the middle (an agent on the critical path) — yielding consistency, fairness, and auditability without the SPOF, bottleneck, and domain coupling. These three choices reinforce one another: because there is no single blended score (each peer keeps its own objective, 6.1), nothing needs an agent to combine scores; because the rules live in a shared Policy Store (§10.3), nothing needs an agent to hold them in the middle. Operational metrics tune the policy over time; they never decide an individual case in the moment.

*1 — Direct T-800 call (no SKYNET on the happy path). The workflow calls T-800 directly because T-800 holds the authoritative daily availability (post-plan inventory/ATP). To confirm with dev: T-800 must own this data as source-of-truth at request time, and the cross-side MCP call (demand-side workflow → supply-side sub-agent) is an accepted invocation (home-side ownership ≠ caller-side invocation).

*2 — Hard-lock must be atomic. This is an implementation guarantee, not a diagram change. Confirm T-800 hard-lock is atomic under concurrency.

*3 — SKYNET earns its place only on escalation. When T-800 can commit, SKYNET adds nothing and is skipped. Open: if profiling later shows the extra A2A hop on escalation is too slow, the Tier-1 read-only probe could also be made a direct call; the Layer-A/B boundary would not otherwise move.

11.2 Deterministic Tiebreak Order

Applied top-down; the first rule that separates two options wins.

  1. Hard constraint — infeasible options are never selected.
  2. CBS tier — Black > White > Spot.
  3. Commitment type — Most / contract > Bidding / opportunistic.
  4. On-time for higher tier > lower cost-to-serve.
  5. Earlier ATP > later ATP.
  6. Lower cost-to-serve > higher (final tiebreak).

12.1 Why this matters

The biggest cost of multi-agent systems isn't building agents — it's integrating them. By choosing A2A as the peer protocol, the cost of adding partner #N is near-zero: publish an Agent Card, exchange OAuth credentials, schema-match the FEAS payload. Compare to a custom protocol that would require bespoke integration with every partner.

"Imagine your CRM agents coordinating with your vendor's supply chain agents — no human middleware, just structured handoffs." — Salesforce 2026 roadmap. This is what Phase 4 looks like. Phoenix is the foundation.

§ 13

FAQ

Quick answers to the questions this design draws most often, each pointing to the section that treats it in full. Three themes: the core model, how sub-agents fit together, and what happens when things fail.

13.1 Core Model — Layers & Orchestration

QuestionShort answerSee
What is the difference between Layer A and Layer B, and how do I decide which one a task belongs in?Layer A = fixed-sequence pipelines on a workflow engine (deterministic, no LLM); Layer B = runtime-decided orchestration by JARVIS/SKYNET. Run the six-question test — fixed order + structured data + no mid-flow human ⇒ Layer A; otherwise Layer B.§3.2, §3.3
Why doesn't the happy path go through an orchestrator? Isn't "through the orchestrator" slow?Cost/latency come from invoking an LLM, not from passing through logic. The happy path is a workflow engine calling sub-agents over MCP directly — no orchestrator, no LLM. The orchestrator engages only on escalation.§3.7, FIG 8.1
What's the difference between JARVIS and SKYNET — who decides what?Two objectives, not one shared score. SKYNET decides feasibility and can veto (ATP/cap/zone/credit); JARVIS decides desirability (price/tier/SQ/date/margin). Hard constraints beat soft preferences, resolved lexicographically.§7.1
If the two sides disagree, who wins? Why is there no central arbiter?There is no referee by design. Conflicts resolve through a shared contract (hard > soft) and a deterministic, reproducible tiebreak order; a human is the exception path only when no feasible option exists.§7.5, §11.2
When does SKYNET (or SIGMA) actually get involved?Only on escalation. SKYNET earns its place when a commitment must be re-planned/displaced or a human must decide. SIGMA runs only on Tier-2 replan — the happy path's daily availability lives in T-800.§6.2, FIG 8.1 *3

13.2 Sub-Agents — How They Fit Together

QuestionShort answerSee
If agents don't call each other directly, how does each agent know what to send and what to receive?Three mechanisms: every capability declares an input/output schema (the shape); the caller (workflow or orchestrator) owns the order and composes one agent's output into the next agent's input; shared facts are read from Data Products / events. An agent only needs to know its own schema.§9.3, §4.1, §3.6
Who guarantees the data passed between agents is complete?Two kinds. Structural completeness (required fields/types) is enforced at the MCP boundary — a bad call fails fast. Business completeness (no demand row or quantity lost) is the caller's job, checked with conservation rules like SIGMA's Σ Load Qty == SQ/RA.§9.3, §4.5
Does an agent need to know which agent ran before or after it?No. Agents are callees that don't care who invoked them; sequencing lives entirely with the caller. That is exactly why "no agent calls another agent" works.§3.6, §9.3
What is "home side" vs "caller side"? Can a JARVIS-side agent be called by SKYNET (and vice-versa)?Home side = where an agent is owned; it does not restrict who may invoke it. Cross-side invocation is allowed and expected (a workflow may call a SKYNET-side agent over MCP).§4.0, §4.3
How do I add a new sub-agent to Phoenix?Follow the recipe: pick the layer, publish an Agent Card + MCP server with capability schemas, set home side & caller policy, register any hard constraint in the Policy Store, wire its state/events, then list it in the roster with the standard failure envelope.§4.4, §4.0

13.3 Failure & Edge Cases

QuestionShort answerSee
If ATLAS cannot book a truck/vessel for the data it was given, what should the system do — solve it as a workflow, or escalate to SKYNET?Three tiers. Tier-1: ATLAS self-heals in Layer A if the fix stays inside the committed load window (other slot/carrier/mode). Tier-2: if the only fix moves the load date or displaces another order, it raises a deviation and SKYNET re-sequences. Tier-3: HITL signs off when the fix hurts another customer's commitment. SARAH and JOHN follow the same shape.§8.5, §4.5
An MCP call times out or a sub-agent is down — who is responsible for retry?The caller. Bounded timeout, retry with backoff, idempotency key on writes; async paths are at-least-once with consumer dedupe; a missed FEAS deadline cancels the task and releases reservations.§4.5
What happens when T-800 cannot commit the requested order?The workflow stops the happy path and escalates to JARVIS, which opens an A2A FEAS exchange; SKYNET orchestrates a Tier-1/Tier-2 replan and returns options (COUNTER) or a confirmation.FIG 8.1 (10b), §8.2, §8.4
What stops two concurrent inquiries from committing the same inventory?The commit/hard-lock serialises in T-800 — one inquiry wins, the other queues or fails. Hard-lock atomicity under concurrency is a stated design condition.FIG 8.1 *2, §4.5
The ATP we promise the customer — is that T-800's or ATLAS's number?Two stages. T-800 gives a provisional ATP at commit ("ready per plan, slot exists") — enough to close the deal but not final. ATLAS produces the firm ATP near load time when it resolves the date range and books the actual carrier. T-800 alone cannot final-confirm; the real load day is execution, owned by ATLAS.§8.6, §11.5
Domestic orders often open today and load tomorrow — do we have to firm the route up front?Yes — that's Mode B (inline). With no buffer to fix a booking later, the workflow adds one Layer-A hop to ATLAS at commit to check carrier availability, so a "can't ship" surfaces before we promise, not after. Long-lead normal export stays Mode A (deferred); the test is atlas_booking_lead vs the load horizon, not the channel.§8.6
